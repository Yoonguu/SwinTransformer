Swin Encoder: 짝을 이루는 블록 구조 (Shifted Window Logic)

Swin Transformer의 인코더는 모델은 depths=(2, 2, 2, 2) 안의 depth=2 설정에 따라 한 스테이지에서 두 번의 연산을 수행합니다.
이는 윈도우 경계를 가로질러 "한 번은 정직하게, 한 번은 어긋나게"정보를 섞어서 데이터를 보기 위한 필수 설계입니다.

            # bi(block index)가 0(짝수)이면 일반 윈도우, 1(홀수)이면 Shifted 윈도우
            for bi in range(depth):
                shift = (0, 0, 0) if (bi % 2 == 0) else tuple(w // 2 for w in self.window_size)

                # blocks에 추가되는 순서: [W-MSA 블록] -> [SW-MSA 블록]
                blocks.append(SwinBlock3D(dim=dim, shift_size=shift, ...))

> 🧐 왜 2개가 한 세트인가요?
  • Window Size: "어디까지 한꺼번에 볼 것인가?" (창문의 크기)
  • Shift Size: "경계선을 어떻게 없앨 것인가?" (창문을 미는 거리)

  • Block 0는 윈도우 내부만 보기 때문에 경계 밖의 정보는 알 수 없습니다.
  • Block 1은 윈도우를 roll시켜 이전 경계선을 윈도우의 중심으로 가져옵니다.
  • 결과적으로 2개 블록([고정 어텐션] + [밀어서 어텐션])을 통과하면 3D 이미지 전체의 모든 영역의 정보가 완벽히 섞여 유기적으로 연결된 특징을 갖게 됩니다.

🔍 연산 단계별 의미:
  • Shortcut 저장: 입력 데이터 `x`를 `shortcut` 변수에 복사해둡니다. 이는 나중에 Residual Connection (더하기 연산)을 통해 기울기 소실을 방지하는 데 사용됩니다.
  • Layer Normalization: `self.norm1(x)`는 각 토큰의 특징(C) 채널에 대해 정규화를 수행하여 학습의 안정성을 높입니다.

패딩 연산 (Padding for Window Partition)

Stage 0 해상도가 윈도우 크기 (2, 7, 7)의 배수가 되도록 부족한 만큼 살을 붙입니다. (※ _build_attn_mask 밖에서 구현됨)

축 (Axis) 원본 해상도 윈도우 크기 계산 로직 최종 해상도 (Padded) 추가된 패딩
Depth (D) 91 2 91 + 1 = 92 92 +1
Height (H) 91 7 91 + 0 = 91 91 0
Width (W) 109 7 109 + 3 = 112 112 +3
💡 왜 패딩을 하나요?

윈도우 분할 시 데이터가 남거나 모자라지 않게 정확한 배수를 맞추기 위함입니다.
결과적으로 3D 볼륨은 (92, 91, 112) 크기가 되어, 어떤 위치에서도 (2, 7, 7) 크기의 윈도우로 깔끔하게 나누어집니다.

🔍 연산의 핵심 (Why Roll?):
  • 경계면 이동: 왼쪽 끝에 있던 토큰들(0, 1, 2)을 오른쪽 끝으로 보냄으로써, 평소에는 만나지 못했던 윈도우 경계의 데이터들이 서로 인접하게 됩니다.
  • 순환 구조: 데이터가 손실되지 않고 한 바퀴 돌아 제자리를 찾는 특징이 있어 Cyclic Shift라고도 불립니다.
  • 주의: 이 연산 후에는 반드시 Attention Mask를 적용하여, 물리적으로는 가까워졌으나 실제 공간적으로는 멀리 떨어져 있는 데이터 간의 잘못된 연산을 차단해야 합니다.
