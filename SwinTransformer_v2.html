📍 구조적 해석:
1. 938,224개의 모든 복셀은 3D 슬라이싱 규칙에 따라 0~26번 사이의 '고향 주소'를 가집니다.
2. 이 거대한 덩어리를 9,568개의 방(Window)으로 조각냅니다.
3. 각 방 안에는 98명이 살게 되며, 이들의 주소를 대조하여 98x98 마스크를 만듭니다.
3단계: window_partition_3d: 지도를 조각내기

패딩된 최종 볼륨 (92, 91, 112)을 기준으로 윈도우를 쪼갭니다.

어떻게 9568개가 나오나요? 9,568 nW

(92 / 2) × (91 / 7) × (112 / 7)
= 46 × 13 × 16 = 9,568

방 하나에는 몇 명이 사나요? 98 Tokens

윈도우 크기 2(D) × 7(H) × 7(W) = 98개

Swin-UNETR: 3D Window ($7 \times 7 \times 2$) 배치 예시

가운데 4x4 구역이 유지되고 가장자리에 1~2칸의 조각이 붙은 형태

Layer 1 (Top Side)

Slice: [0:-1]

Layer 2 (Bottom Side)

🧐 3D 윈도우($7 \times 7 \times 2$)의 데이터 흐름:
1. 입체적 구역 분리: 윈도우는 $7 \times 7$ 판데기 두 개가 겹쳐진 형태입니다. Depth 축 Shift(1)로 인해 위층(0~8)과 아래층(9~17)의 주소가 완전히 분리됩니다.
2. 가장자리 조각들 (Zone 0, 1, 2, 3, 5, 6, 7, 8): 이들은 윈도우 경계에 걸쳐서 반대편 끝에서 날아와 붙은 1~2칸짜리 이물질들입니다.
3. 수치 정정 반영: 총 938,224개의 복셀은 이와 같은 방식으로 0~26번의 주소를 할당받으며, 그중 98개씩 묶여 9,568개의 윈도우가 됩니다.
4. 4x4 중심의 의미: 각 층의 중앙(4번, 13번)은 연속성이 보장된 데이터이며, 이들끼리는 층이 달라도 주소 차이가 발생하므로 어텐션 마스크가 "같은 층, 같은 구역"인 토큰끼리만 대화를 허용합니다.

1. 9568 (윈도우의 개수)

9,568 nW
- 의미: 전체 3D 공간을 쪼개 만든 독립된 방의 총 개수입니다.
- 계산: (92/2) × (91/7) × (112/7) = 9,568개
- 역할: GPU가 동시에 처리하는 배치(Batch) 크기로 사용됩니다.

2. 98 (방 안의 사람 수)

98 Tokens
- 의미: 하나의 윈도우(방) 안에 살고 있는 픽셀(Voxel)의 수입니다.
- 계산: 2(D) × 7(H) × 7(W) = 98개
- 역할: 어텐션 행렬의 시퀀스 길이가 되어 98 x 98 관계도를 만듭니다.
mask_windows = window_partition_3d(img_mask, self.window_size)
# [1, 92, 91, 112, 1] -> [9568, 2, 7, 7, 1]
# 텐서 모양 변화 과정
[1, 92, 91, 112, 1] ➔ [9568, 2, 7, 7, 1] ➔ [9568, 98]
의미: 3D 공간 전체에 그려진 27개 구역의 번호판(img_mask)을 실제 연산 단위인 윈도우(Window) 크기로 자르는 과정입니다.
  • 코드 로그 기준, 92x91x112 크기의 거대 지도를 2x7x7=98 크기의 작은 블록들로 나눕니다.
  • 결과적으로 9,568개의 독립된 윈도우 조각이 생성됩니다.
  • 각 방 안에는 98명의 사람이 있고, 이들은 오직 방 안에서만 서로 인사(98x98 어텐션)를 나눕니다.
  • 따라서 전체 연산량은 [9568개 방] × [98 x 98 관계 계산]이 되며, GPU는 이 거대한 양을 병렬로 동시에 처리하고 있는 것입니다.
4단계: unsqueeze(1) - unsqueeze(2) (실제 주소 대조)

9568개의 각 윈도우 내부에서 98x98 관계 지도를 생성합니다.
9568개의 방 각각에서, 98명의 토큰들이 "너랑 나랑 고향(구역 번호)이 같니?"를 대조합니다.

# [9568, 1, 98] - [9568, 98, 1] ➔ [9568, 98, 98]
# unsqueeze(1): [9568, 1, 98] - 가로로 눕히기
# unsqueeze(2): [9568, 98, 1] - 세로로 세우기
attn_mask = mask_windows.unsqueeze(1) - mask_windows.unsqueeze(2)

5단계: masked_fill (물리적 차단벽 -100.0 설치)

계산된 차이값을 바탕으로, 어텐션이 일어나면 안 되는 곳에 강력한 페널티를 부여합니다.

# 0이 아닌 곳(가짜)은 -100.0으로, 0인 곳(진짜)은 0.0으로!
attn_mask.masked_fill(attn_mask != 0, float(-100.0)).masked_fill(attn_mask == 0, 0.0)

# 1. 주소 차이가 0이 아닌 곳(가짜 이웃)을 찾아 -100.0으로 덮어쓰기
attn_mask = attn_mask.masked_fill(attn_mask != 0, float(-100.0))

# 2. 주소 차이가 0인 곳(진짜 이웃)을 찾아 0.0으로 덮어쓰기
attn_mask = attn_mask.masked_fill(attn_mask == 0, 0.0)
함수 설명: masked_fill(mask, value)는 조건(mask)이 True인 위치의 값들을 특정 값(value)으로 덮어쓰는 파이토치 함수입니다.
  • 첫 번째 fill: 주소 차이가 0이 아닌 곳(가짜 이웃)을 -100.0으로 바꿉니다.
    • attn_mask != 0: "고향이 다른 토큰 쌍들을 모두 찾아라!"라는 조건문입니다.
    • float(-100.0): 가짜 이웃들의 어텐션 점수를 강제로 깎아내리기 위한 값으로 주소 차이가 있는 가짜 이웃들에게 -100.0을 주어 대화를 차단합니다. 나중에 Softmax를 거치면 이 값은 0(확률 없음)이 됩니다.
  • 두 번째 fill: 주소 차이가 0인 곳(진짜 이웃)을 0.0으로 바꿉니다.
    • attn_mask == 0: "원래 이웃이거나 같은 구역에서 온 진짜 이웃들을 찾아라!"라는 조건문입니다.
    • 0.0: 같은 구역 출신인 진짜 이웃들의 어텐션 점수에는 아무런 영향을 주지 않도록 중립 값인 0.0을 주어 정상 어텐션을 허용합니다.
결과: 이 마스크를 어텐션 점수에 더하면, -100.0이 된 관계는 softmax 계산 시 확률이 0이 되어 정보 교환이 완벽히 차단됩니다.
